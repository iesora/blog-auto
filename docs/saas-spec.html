1. 概要と前提

現行の blog-auto は 1社1WordPressサイト専用 の運用ツールとして作られている。 これを複数顧客が利用できるSaaSへ拡張するため、テナント分離・課金・セルフサービス・運用基盤を追加する。

SaaS化のゴール

  • 顧客がサインアップから 30分以内 に最初のドラフト記事を生成できる
  • 複数のWordPressサイト・複数ユーザーを 1テナント内で安全に分離 しつつ管理できる
  • 使用量に応じた 従量課金 + 月額サブスク を成立させる
  • 顧客のキーワード/ブランドガイドを学習して 記事の質をテナント単位で最適化

想定ターゲット

セグメント規模主要ユースケース
個人ブロガー / アフィリエイター1サイト・月10〜30本SEO記事の量産、サムネ自動化
中小ECサイト1〜3サイト・月30〜100本商品紹介・ランキング・FAQ
Web制作会社・代理店クライアント数10〜50クライアントごとの代理運用、レポート

2. 現状アーキテクチャ

[Cloud Scheduler] ──OIDC──▶ [Cloud Run: NestJS] │ ├─▶ Anthropic API (Claude 4.6 で記事生成) ├─▶ Gemini API (画像生成) ├─▶ WordPress REST (固定の1サイトに投稿) └─▶ GCS (schedules.xlsx を直接読み書き)

現状の制約

レイヤ制約SaaS化での問題
認証Basic認証 1組複数顧客を分離できない
WordPress接続.env に1サイト固定顧客ごとの資格情報を持てない
スケジュール保存GCS上の単一Excel同時編集・テナント分離不可、ロールバック不能
ジョブ実行Cloud Schedulerが「今日の1件」を叩く顧客ごとの並列実行・リトライ制御ができない
APIキーAnthropic/Gemini が共有使用量を顧客に按分できない、コスト爆発リスク
UI静的 public/ + Basic認証サインアップ・課金画面が無い

4. マルチテナンシー

4.1 データモデルの基本方針

シングルDB / シェアードスキーマ + tenant_id 行レベル分離 を採用する。理由:

  • SMB中心ターゲットでテナント数が多くなる想定 → スキーマ分離はマイグレーション運用コストが破綻する
  • すべてのクエリに WHERE tenant_id = ? を強制するため、ORM層（Prisma/TypeORM）でテナントスコープのミドルウェア を必須化する
  • 将来エンタープライズ顧客向けにDBインスタンス分離プランを別途用意できる余地を残す

4.2 階層

Organization (= 課金単位 / テナント) ├─ Members (User × Role) ├─ Sites (WordPress接続) │ └─ Schedules / Articles / RunHistory ├─ BrandProfiles (記事生成設定) ├─ ApiKeys └─ Subscription (Stripe)

4.3 ロール

ロール権限
Owner請求・組織削除・全操作
Adminサイト/メンバー管理・設定変更（請求は不可）
Editor記事生成・スケジュール作成・公開
Viewer閲覧のみ（レポート用）

5. 認証・アカウント

5.1 サインアップフロー

  1. Email + パスワード入力 → 確認メール送信
  2. メール内リンクで verified=true
  3. 初回ログイン → Organization作成（社名 / 個人名）
  4. WordPress接続ウィザード（後述）
  5. 無料トライアル開始（クレカ登録不要・3記事まで）

5.2 認証方式

  • セッション: HttpOnly Cookie + CSRFトークン (UI用)
  • API: X-Api-Key ヘッダ（Bearerでも可）
  • SSO: Google OAuth (Phase 1), SAML (Phase 3 / Enterprise)
  • 2FA: TOTP (Authenticator App)

5.3 既存実装からの移行

既存 Basic認証 / OIDC ガードの扱い

  • scheduler/oidc-auth.guard.ts: Cloud Scheduler→Cloud Run の内部呼び出し用に残す（システム間通信のみ）
  • main.ts のBasic認証: 廃止し、セッション認証ミドルウェアに置換
  • 運用者用の Admin Console は別ドメイン（admin.example.com）で IPアロー + SSO

6. WordPressサイト管理

6.1 接続ウィザード

  1. サイトURL入力 → REST APIエンドポイントの存在確認 (/wp-json/)
  2. 認証方式選択:
    • Application Password（推奨）: ユーザー名 + 16桁パスワード
    • OAuth2 プラグイン経由（将来）
  3. 接続テスト: 投稿一覧取得 + メディアアップロード権限の確認
  4. カテゴリ・タグの初回同期（DB側にミラー）

6.2 資格情報の保存

  • WordPressのアプリケーションパスワードは KMS（GCP Cloud KMS / AWS KMS）で暗号化 してDBに保存
  • 復号は実行時にメモリ上のみ。ログ出力禁止のフィルタを ConfigService にかける
  • キーローテーション: 90日ごとに envelope encryption の DEK を再生成

6.3 ヘルスチェック

  • 1日1回、各サイトに対して軽量な GET /wp-json/wp/v2/users/me を実行
  • 連続3回失敗 → ダッシュボードで警告 + メール通知
  • パスワード期限切れ・サイトダウン・SSL切れを区別したエラーコード

8. 使用量・レート制限

8.1 メータリング

  • 記事生成完了時に UsageRecord(tenantId, type='article', cost=1, ts) を書き込む
  • 画像は type='image' で別カウント（プランによる）
  • 月初にリセット。プラン変更時はProrationを反映

8.2 制限のかけ方

制限実装
月次クォータ記事生成APIの入口で UsageService.canConsume(tenant, 'article') を確認
同時実行数キューのテナント別concurrency（Free:1, Pro:5, Agency:20）
APIレートリミットRedis + sliding window。プラン別RPM
サーキットブレーカWordPress接続が連続失敗する顧客を一時的に停止

9. ジョブ実行基盤

9.1 アーキテクチャ移行

[UI / API] ──▶ [enqueue: Cloud Tasks or BullMQ] │ ▼ [Worker (Cloud Run jobs / 別サービス)] ├─ Claude生成 ├─ Gemini画像生成 ├─ WordPressアップロード └─ Webhook通知 + UsageRecord

9.2 スケジューラの再設計

現行は「全顧客の今日の予約を1つのCloud Schedulerが叩く」が、SaaSでは破綻するため:

  • マスターcron（毎時0分）が Schedule テーブルから直近1時間に発火すべきジョブをSELECT → Cloud Tasks に投入
  • ユーザーは UI でカレンダー / cron式（簡易プリセット）を指定
  • タイムゾーンはテナント設定に従う（デフォルト: Asia/Tokyo）

9.3 既存 SchedulerStorageService の置換

Excel-on-GCS の廃止

  • 現状 schedules.xlsx 単一ファイル → 同時実行で破損リスク
  • Postgres の schedules テーブルに完全移行
  • ユーザーへの「Excel一括取り込み」UXは残す（CSV/Excelインポート機能として）

10. ブランド設定・テンプレート

10.1 BrandProfile

  • 会社/ブログ名、業種、想定読者ペルソナ
  • 文体: 敬体 / 常体 / 砕けた口語 / プロフェッショナル
  • NGワード（薬機法対応など）・必須CTAフレーズ
  • 内部リンクのドメインルール

10.2 プロンプトテンプレート

現行の prompt-templates.ts の buildPrompt(articleType, keywords, topic) を拡張:

buildPrompt({
  articleType,
  keywords,
  topic,
  brandProfile,    // 追加
  customSystemPrompt, // Pro以上で編集可
  examples,        // few-shot: 過去の高評価記事
})

10.3 記事タイプの拡張

現行の SEO / REPAIR_REPORT / QA / RANKING に加え、ユーザー定義テンプレを許容（Agencyプラン以上）。

12. 運用・監視

12.1 メトリクス

  • テナント別: 記事生成成功率、生成時間 p50/p95、AI APIエラー率、WordPress 5xx率
  • システム全体: キュー滞留、Claude/Geminiのレイテンシ、コスト/日
  • ツール: Cloud Logging + Cloud Monitoring + Sentry（フロント/バック共通）

12.2 監査ログ

すべての書き込み操作を audit_logs に記録（actor, tenantId, action, target, ip, userAgent, diff）。Owner権限で90日分閲覧可能。

12.3 アラート

  • テナント別エラー率 > 20% / 1時間 → 顧客にメール通知
  • キュー滞留 > 100件 → PagerDuty
  • AI原価が日次予算の150% → Slack通知 + 自動レート制御

13. セキュリティ・コンプライアンス

13.1 セキュリティ要件

  • WordPress資格情報・APIキーは KMS 暗号化必須
  • 監査ログ・パスワードハッシュ（Argon2id）
  • OWASP Top 10 対応（XSS / CSRF / SSRF対策）。特にWordPressサイトURLは SSRFブロックリスト（プライベートIP, metadata endpoint 等）必須
  • 依存パッケージの Dependabot / Renovate 自動更新
  • ペネトレーションテストを年1回（外部）

13.2 法務

  • 利用規約 / プライバシーポリシー / 特商法表記（日本向け必須）
  • データ削除リクエスト（30日以内対応）
  • GPT/Claudeの出力に対する免責条項（薬事/医療/金融分野の利用注意）
  • サブプロセッサ一覧の公開（Anthropic, Google, Stripe, GCP, etc）

17. フェーズ別ロードマップ

18. 主要リスクと対策
