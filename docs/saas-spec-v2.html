1. 概要 / ゴール

既存の blog-auto（NestJS + Claude + WordPress + GSC + GCS Excel）を 5サイト同時運用 と キーワードの自動生成 に拡張し、 永続化層を GCS 上の Excel から MySQL + TypeORM に置き換える。

到達したい状態

  • 毎日、登録された 5つの WordPress サイト に 1本ずつ下書きが投稿される
  • キーワードは 28日に1度、GSC データをもとに Claude が自動生成 し、向こう28日分のスケジュールに自動投入
  • 運用者は 承認 UI で keyword を上書き / 拒否 できる（フェイルセーフ）
  • データはすべて MySQL に集約、TypeORM の Entity / Migration で管理

非ゴール（v2 では扱わない）

  • WordPress 以外の CMS（Ghost / Wix など）
  • 投稿の自動公開（v2 までは status=draft のまま）
  • 多言語化（日本語のみ）
  • 本格的なマルチテナント SaaS 化（テナント分離は v3 以降）

キーポイント

  • 1 Cloud Run サービス / 1 MySQL インスタンスに集約（マルチテナント化は後段）
  • Cloud Run → Cloud SQL は Cloud SQL Auth Proxy / Private IP 経由

Site

// src/sites/site.entity.ts
@Entity({ name: 'sites' })
export class Site {
  @PrimaryGeneratedColumn() id!: number;

  @Column({ unique: true, length: 64 })  slug!: string;          // 'gakkiou', ...
  @Column({ length: 128 })                name!: string;

  @Column({ name: 'wp_url' })             wpUrl!: string;
  @Column({ name: 'wp_username' })        wpUsername!: string;
  @Column({ name: 'wp_app_pw_encrypted', type: 'text' })
                                          wpAppPwEncrypted!: string;  // KMS で暗号化

  @Column({ name: 'gsc_site_url' })       gscSiteUrl!: string;
  @Column({ name: 'default_article_type', type: 'enum', enum: ArticleType,
            default: ArticleType.SEO })   defaultArticleType!: ArticleType;
  @Column({ name: 'default_categories', type: 'json', nullable: true })
                                          defaultCategories?: string[];
  @Column({ name: 'default_tags', type: 'json', nullable: true })
                                          defaultTags?: string[];

  @Column({ type: 'text', nullable: true }) persona?: string;   // Claude 用のサイト人格
  @Column({ default: true })              active!: boolean;

  @CreateDateColumn({ name: 'created_at' }) createdAt!: Date;
  @UpdateDateColumn({ name: 'updated_at' }) updatedAt!: Date;

  @OneToMany(() => ScheduleEntry, (s) => s.site)  schedules!: ScheduleEntry[];
  @OneToMany(() => KeywordPlan,   (p) => p.site)  keywordPlans!: KeywordPlan[];
}

ScheduleEntry

// src/scheduler/schedule-entry.entity.ts
export type ScheduleStatus = 'pending' | 'approved' | 'skipped';
export type ScheduleSource = 'manual' | 'auto';

@Entity({ name: 'schedule_entries' })
@Unique(['site', 'scheduledDate'])     // 1サイト × 1日 = 1エントリ
export class ScheduleEntry {
  @PrimaryGeneratedColumn() id!: number;

  @ManyToOne(() => Site, (s) => s.schedules, { onDelete: 'CASCADE' })
  @JoinColumn({ name: 'site_id' })       site!: Site;

  @Column({ name: 'scheduled_date', type: 'date' })  scheduledDate!: string; // 'YYYY-MM-DD'

  @Column({ type: 'enum', enum: ['pending', 'approved', 'skipped'],
            default: 'pending' })        status!: ScheduleStatus;

  @Column({ length: 100 })                keyword1!: string;
  @Column({ length: 100 })                keyword2!: string;
  @Column({ length: 100 })                keyword3!: string;

  @Column({ type: 'text', nullable: true })   topic?: string;
  @Column({ name: 'article_type', type: 'enum', enum: ArticleType, nullable: true })
                                          articleType?: ArticleType;
  @Column({ name: 'category_names', type: 'json', nullable: true })
                                          categoryNames?: string[];
  @Column({ name: 'tag_names', type: 'json', nullable: true })
                                          tagNames?: string[];
  @Column({ name: 'inline_image_count', type: 'tinyint', nullable: true })
                                          inlineImageCount?: number;

  @Column({ type: 'enum', enum: ['manual', 'auto'], default: 'manual' })
                                          source!: ScheduleSource;

  @ManyToOne(() => KeywordPlan, { nullable: true, onDelete: 'SET NULL' })
  @JoinColumn({ name: 'plan_id' })       plan?: KeywordPlan;

  @OneToMany(() => RunHistory, (r) => r.scheduleEntry) runs!: RunHistory[];

  @CreateDateColumn({ name: 'created_at' }) createdAt!: Date;
  @UpdateDateColumn({ name: 'updated_at' }) updatedAt!: Date;
}

GscSnapshot / GscQueryRow

// src/search-console/gsc-snapshot.entity.ts
@Entity({ name: 'gsc_snapshots' })
export class GscSnapshot {
  @PrimaryGeneratedColumn() id!: number;
  @ManyToOne(() => Site, { onDelete: 'CASCADE' })
  @JoinColumn({ name: 'site_id' })  site!: Site;
  @Column({ name: 'start_date', type: 'date' }) startDate!: string;
  @Column({ name: 'end_date',   type: 'date' }) endDate!: string;
  @Column({ name: 'row_count' })                rowCount!: number;
  @Column({ name: 'taken_at', type: 'datetime', default: () => 'CURRENT_TIMESTAMP' })
                                                takenAt!: Date;
  @Column({ name: 'data_state', length: 16, default: 'final' }) dataState!: string;

  @OneToMany(() => GscQueryRow, (r) => r.snapshot) rows!: GscQueryRow[];
}

@Entity({ name: 'gsc_query_rows' })
@Index(['snapshot', 'impressions'])
export class GscQueryRow {
  @PrimaryGeneratedColumn() id!: number;
  @ManyToOne(() => GscSnapshot, (s) => s.rows, { onDelete: 'CASCADE' })
  @JoinColumn({ name: 'snapshot_id' })  snapshot!: GscSnapshot;

  @Column({ length: 255 })                    query!: string;
  @Column({ length: 512, nullable: true })    page?: string;
  @Column()                                   clicks!: number;
  @Column()                                   impressions!: number;
  @Column({ type: 'float' })                  ctr!: number;
  @Column({ type: 'float' })                  position!: number;
}

6. 自動キーワード生成（28日サイクル）

6.1 サイクル定義

  • サイクル長: 28日（GSC の表示単位「過去28日」と一致）
  • トリガ: 月1回、Cloud Scheduler の plan-next-cycle ジョブ
  • 対象期間: 「明日」から「明日 + 27日」までの 28日 × 5サイト = 140エントリ
  • サイクル境界: 各 KeywordPlan.cycleStart / cycleEnd で表現（半開区間ではなく両端含む）

6.2 フロー

[1] Cloud Scheduler → POST /keywords/plan-next-cycle │ ▼ [2] 全 active サイトを sites から取得 (5件) │ ▼ サイトごとに並列 [3] SearchConsoleSvc.fetchAndStore(site) - GSC /searchAnalytics/query (dims=['query','page'], 直近28日) - INSERT gsc_snapshots / gsc_query_rows │ ▼ [4] KeywordPlannerSvc.generatePlan(site, snapshot) - スナップショットから候補シードを抽出: A. impressions ≥ 100 かつ ctr ≤ 1.0% のクエリ B. position が 8〜20 の改善余地クエリ C. 直近28日に投稿済みの記事と被らないクエリ - Claude に投げて: * 28日分 × 3キーワード を JSON で返させる * 各日に topic / articleType / category も提案させる │ ▼ [5] INSERT keyword_plans (status='draft', raw_response=...) + INSERT schedule_entries (status='pending', source='auto', plan_id=...) ※ 既存 status='approved' のエントリは上書きしない │ ▼ [6] 通知（Slack / Email）：レビュー URL を送付 │ ▼ [7] 運用者が UI で承認 → keyword_plans.status='approved' → 配下の schedule_entries.status='pending' → 'approved' に一括更新

6.3 Claude へのプロンプト要点

あなたは {site.name}（ペルソナ: {site.persona}）の SEO 担当です。
直近28日の Google Search Console データを以下に示します。
これをもとに「次の28日間で書くべきブログ記事」のキーワードを 28 件提案してください。

# 入力データ
- 高インプ低 CTR クエリ Top 50:
  - {query, impressions, ctr, position} × N
- 順位 8〜20 の改善余地クエリ Top 50:
  - ...
- 直近28日に投稿済みの記事タイトル:
  - ...

# 制約
- 1日 = 3つのキーワード（メイン + サブ × 2）
- 既存記事との重複を避け、内部リンク候補となる関連トピックを優先
- 28日分の article_type 分布: SEO 60%, QA 25%, RANKING 10%, REPAIR_REPORT 5%

# 出力（厳密な JSON）
{
  "days": [
    { "offset": 0, "keywords": ["...", "...", "..."], "topic": "...",
      "articleType": "seo", "categoryNames": ["..."], "tagNames": ["..."] },
    ... (28件)
  ]
}

fail-safe ルール

  • Claude のレスポンスが 28件に満たない / JSON 破損 → KeywordPlan.status='draft' のまま即座に Slack 通知。schedule_entries は作らない。
  • 同じキーワードが28日内で 2 回以上出現 → 警告ログ、運用者が手動修正できるよう raw_response を必ず保存。
  • plan-next-cycle は冪等。同じ cycleStart で2度走っても UNIQUE(site_id, cycle_start) に追加して、新規プランを生成（既存プランは draft のまま残す or 上書きフラグで制御）。

7. 日次実行フロー（5サイト同時投稿）

7.1 シーケンス

Cloud Scheduler (09:00 JST) ──POST /schedules/run-today──► SchedulerController │ ▼ repo.find({ scheduledDate: today, status: 'approved' }) │ ▼ for each (最大5件) ┌────────────────────────────────────────┴───────────────┐ │ entry#1: site_A ... entry#5: site_E │ │ BlogGeneratorSvc.generateAndCreateDraft({site, ...}) │ │ ├─ Claude (本文) │ │ ├─ 画像生成 × N (並列) │ │ ├─ WP.uploadMedia (siteごとの認証) │ │ └─ WP.createPost (siteごとの認証) │ │ INSERT run_history(status, wp_post_id, duration_ms) │ └────────────────────────────────────────────────────────┘ │ ▼ レスポンス: { processed: 5, succeeded, failed }

7.2 並列度と分離

  • 5サイト分は Promise.allSettled で並列。1サイト失敗が他に波及しない。
  • Claude API のレート制限を考慮し、内部で p-limit(concurrency=3) 程度に絞る。
  • WP 投稿は サイトごとの認証情報 を都度復号して使い、終わったらメモリから破棄。

7.3 リトライ

  • 1次リトライ: Cloud Scheduler のリトライ機能（最大3回、指数バックオフ）
  • 2次リトライ: POST /schedules/run/:date を手動で叩く（冪等：成功済みエントリはスキップ）
  • 成功済みかどうかは「最新の run_history.status='success'」で判定

8. Excel → MySQL 移行

8.1 マイグレーションスクリプト

// scripts/migrate-from-excel.ts
//   既存 GCS 上の schedules.xlsx を読み、デフォルト site (slug='gakkiou') に紐づけて
//   schedule_entries / run_history に流し込む。
//   ※ 既存データは丸ごと「manual / approved」として取り込む。
//
// 使い方:
//   $ npx ts-node scripts/migrate-from-excel.ts --site gakkiou --dry-run
//   $ npx ts-node scripts/migrate-from-excel.ts --site gakkiou --commit

8.2 ステップ

  1. Cloud SQL インスタンス作成 / DB / ユーザ準備
  2. TypeORM Migration を npm run migration:run で適用（空 DB）
  3. 初期 sites レコード 5件を seeds/sites.ts から投入
  4. migrate-from-excel.ts --dry-run で件数 / 警告を確認
  5. --commit で本番反映、終わったら旧 Excel を schedules.xlsx.bak.YYYYMMDD にリネーム
  6. Cloud Run の env を切り替え、SchedulerStorageService 廃止 PR をマージ

8.3 ロールバック

  • 旧 Excel は読み取り専用で残し、移行直後の MySQL ダンプを gs://<backup>/db-pre-cutover.sql.gz に保存
  • 問題発生時は env を 1コミット前に revert（コードは Excel / DB 両対応のまま 1スプリント維持する）

9. Cloud Scheduler 構成

ジョブ名cron呼び出し用途
blog-auto-run-today 0 9 * * * (JST) POST /schedules/run-today 毎日 09:00 に当日分5サイトを実行

※ cron では「28日ごと」は厳密に表現できないため、毎月1日に走らせて cycle_start > today な未来プランがなければ生成、を採る運用が現実的。

12. 運用 / 監視 / リカバリ

監視

  • Cloud Logging で severity=ERROR を Slack に通知
  • 毎朝 run-today 完了後に processed / succeeded / failed サマリを Slack 通知
  • plan-next-cycle の draft 作成も Slack 通知（レビュー URL 付き）

バックアップ

  • Cloud SQL の自動バックアップ（日次、保管7日）
  • 週次で mysqldump --single-transaction を GCS にアップロード

セキュリティ

  • WP Application Password は KMS で暗号化して DB 格納
  • Cloud Run のサービスアカウントは cloudsql.client, cloudkms.cryptoKeyEncrypterDecrypter のみ付与
  • UI は Basic Auth + HTTPS。API は OIDC（Cloud Scheduler）か Basic 認証のみ

リカバリ手順

  1. 日次実行失敗 → POST /schedules/run/YYYY-MM-DD を手動 curl
  2. キーワード生成失敗 → POST /keywords/plan/<slug>
  3. サイトが死亡 → PATCH /sites/<slug> { active: false } で除外

13. リスクと未決事項

未決事項（要オーナー判断）

  1. 5サイトの具体的なドメイン / ペルソナ: 既存 gakkiou.com 以外の4つは未定。slug, persona, default_article_type を埋める必要がある。
  2. 承認フローを入れるか: SHOULD 推奨。スキップする場合は schedule_entries.status を常に 'approved' 直行に。
  3. WP 認証情報の保管方式: (a) KMS で DB 列を暗号化 / (b) Secret Manager に wp-app-pw-<slug> として置き DB には参照名のみ。後者の方が運用が楽。
  4. 1日5本同時投稿の API レート: Claude / Imagen / WP REST の同時実行コスト試算が未。p-limit(3) 程度から開始予定。
  5. 28日サイクルの境界: 月跨ぎや祝日を考慮するか。当面は単純な「cron で月1回 + cycleStart=明日」運用。

主要リスク

  • キーワードの暴走: Claude が誤った / 攻撃的キーワードを生成 → 承認フェーズ必須 / NG ワードフィルタ
  • WP 認証情報漏洩: 5サイト分のパスワードが集約されるため、KMS / Secret Manager と監査ログの取得を厳格に
  • 移行時のデータ消失: 旧 Excel を読み取り専用で 30日保持、移行直後の SQL ダンプを取得
  • Cloud SQL 接続枯渇: Cloud Run のインスタンス × コネクションプール上限を意識（pool: { max: 5 } 程度から）
